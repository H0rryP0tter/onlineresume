1. Browser

User fills in the register or login form on index.html.

The browser sends email and password to the server.

POST /auth/register
POST /auth/login

2. Auth Controller

authController.js validates the request.

Register hashes the password. Login compares the password with the saved hash.

3. User Model

user.js defines users in MongoDB.

It stores email and passwordHash, not the raw password.

4. JWT Created

The server signs a JWT using JWT_SECRET.

The token contains safe user data: id and email.

5. HttpOnly Cookie

The JWT is stored in an authToken cookie.

HttpOnly means frontend JavaScript cannot read the token.

6. Protected Request

When the browser calls a protected route, it sends the cookie automatically.

GET /resume/all

Middleware Check

middleware/auth.js is the gatekeeper. It reads the cookie, verifies the JWT, and decides whether the request can continue.

Valid Token

The middleware sets req.user = { id, email }, then calls next().

Missing or Invalid Token

The request stops immediately with 401 Authentication required or 401 Invalid or expired token.

What Controllers Receive

Browser sends cookie
auth.js verifies JWT
req.user is attached
resumeController.js uses req.user.id

Private Resume Rule

Every resume belongs to one user through userId. The frontend does not decide ownership. The server does.

Resume.find({
  userId: req.user.id
})

Why This Is Secure

If User A tries to access User B's resume, the query still filters by User A's id. MongoDB will not return User B's document.

Resume.findOne({
  _id: resumeId,
  userId: req.user.id
})

Authentication answers: "Who are you?" Authorization answers: "Are you allowed to access this data?" In this project, JWT proves the user identity, and userId: req.user.id enforces access.
